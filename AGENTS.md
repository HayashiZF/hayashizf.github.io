# Repository Guidelines

## Project Structure & Module Organization

This is a Jekyll-powered GitHub Pages site. Root-level `*.html` files are site
pages; shared templates belong in `_layouts/` and `_includes/`. Publish dated
posts in `_posts/` using `YYYY-MM-DD-slug.md` (keep the existing front matter
format). Site settings, metadata, and plugins live in `_config.yml`.

Custom styles start in `less/` and compile to `css/`; edit the LESS source, not
generated CSS, when changing the theme. Custom scripts are in `js/`, with
minified counterparts generated by Grunt. Keep images in `img/` or `img-post/`,
downloadable materials in `files/`, and PWA assets in `pwa/` and `sw.js`.

## Build, Test, and Development Commands

- `npm install` installs the legacy Grunt build dependencies.
- `npx grunt` compiles LESS, minifies the configured JavaScript, and refreshes
  banner comments. Run it after changing `less/` or the project JavaScript.
- `jekyll build` renders the site into `_site/`; this is the CI build check.
- `jekyll serve --watch` runs a local Jekyll server and rebuilds on content or
  template changes. Use a separate terminal for `npx grunt watch` when editing
  LESS or JavaScript.

The `npm run py3view` script can serve a previously generated `_site/` folder
on port 8020. Do not rely on it to rebuild content.

## Coding Style & Naming Conventions

Match surrounding files: four-space indentation in HTML, JavaScript, and LESS;
keep Jekyll/Liquid expressions readable and avoid unrelated reformatting. Use
lowercase, hyphenated asset and post slugs (for example,
`img-post/2026-07-15-example.png`). Preserve minified files only when the
Grunt build regenerates them.

## Testing Guidelines

There is no unit-test framework or coverage target. Before opening a change,
run `jekyll build` and, for theme/script changes, `npx grunt`. Preview affected
pages locally and check desktop/mobile layout, internal links, post front
matter, and any service-worker or PWA behavior changed by the patch.

## Commit & Pull Request Guidelines

Existing history uses short, imperative subjects such as `Update _config.yml`
and `Update page.html`. Follow that pattern, naming the primary area changed.
Keep commits focused. Pull requests should state the user-visible result, list
validation commands run, link relevant issues, and include screenshots for
layout or visual changes. Call out content migrations, configuration changes,
and generated asset updates explicitly.
